# GitHub Copilot: Agent 모드 vs Plan 모드 비교 분석

## 1. 개요 (Overview)
이번 **n8n 워크플로우 고도화 작업**은 **Agent 모드**를 통해 진행되었습니다. 사용자가 "Plan 모드"와의 차이점에 대해 문의하였기에, 이번 작업을 예시로 두 모드의 작동 방식과 차이점을 분석하여 보고합니다.

## 2. Agent 모드 (우리가 작업한 방식)
**"대화형 해결사 (Interactive Solver)"**

*   **작동 방식:** 사용자의 명령을 듣고, 즉시 도구(Tool)를 사용하여 하나씩 해결해 나갑니다. 오류가 발생하면 즉시 수정하고 다시 시도합니다.
*   **이번 작업에서의 모습:**
    1.  사용자: "워크플로우 고쳐줘."
    2.  Agent: "파일 읽어볼게. 아, 구조가 이렇게 되어있네. 수정할게." (수정)
    3.  Agent: "어? 수정하다가 텍스트를 못 찾았네. 다시 시도할게." (재수정)
    4.  Agent: "이제 TTS 단계 추가할게." (추가)
*   **장점:**
    *   **유연성:** 작업 중간에 사용자가 "잠깐, 자막도 추가해줘"라고 요구사항을 바꿔도 즉시 반영 가능.
    *   **디버깅:** 오류가 났을 때 바로 로그를 보고 고치는 데 유리함.
*   **단점:**
    *   **근시안적:** 당장 눈앞의 파일 하나를 고치는 데 집중하느라, 전체 프로젝트에 미칠 영향을 놓칠 수도 있음.

## 3. Plan 모드 (설계 중심 방식)
**"건축가 (Architect & Planner)"**

*   **작동 방식:** 코드를 건드리기 전에 **전체 계획(Plan)**을 먼저 수립합니다. 어떤 파일들을 고쳐야 하고, 각 단계가 서로 어떻게 연결되는지 미리 파악한 후, 사용자의 승인을 받고 한 번에(혹은 단계별로) 실행합니다.
*   **이번 작업을 Plan 모드로 했다면?:**
    1.  사용자: "워크플로우 고도화해줘."
    2.  **Plan 모드:** "전체 프로젝트를 분석했습니다. 목표를 달성하기 위해 3단계 계획을 제안합니다."
        *   *Step 1:* `YouTube_Shorts_Automation_Gemini.json`의 노드 연결 순서 재배치 (직렬화).
        *   *Step 2:* TTS 전용 Agent 노드 정의 및 삽입.
        *   *Step 3:* 자막 생성용 Code 노드 작성 및 Merge 로직 변경.
    3.  사용자: "승인."
    4.  **Plan 모드:** 계획된 순서대로 파일들을 수정 (중간에 멈추지 않고 구조적 변경 수행).
*   **장점:**
    *   **큰 그림:** 여러 파일이 얽혀있는 복잡한 리팩토링에 유리함.
    *   **일관성:** 처음부터 끝까지 논리적인 흐름을 가지고 코드를 작성함.
*   **단점:**
    *   **경직성:** 계획을 세우는 데 시간이 걸리며, 중간에 "아 맞다, 이것도" 하고 끼어들기가 조금 더 번거로울 수 있음.

## 4. 요약 및 결론

| 특징 | Agent 모드 (Current) | Plan 모드 (Alternative) |
| :--- | :--- | :--- |
| **핵심** | **대화 & 즉시 실행** | **분석 & 계획 수립** |
| **비유** | 옆에 앉은 **유능한 개발자** | 전체를 설계하는 **PM/아키텍트** |
| **적합한 상황** | 버그 수정, 기능 추가, 탐색적 코딩 | 대규모 리팩토링, 신규 기능 명세 구현 |
| **이번 작업** | 오류를 만나면 바로 고치며 진행함 | 전체 구조를 먼저 그리고 한 번에 수정했을 것임 |

**결론:** 이번 n8n 작업처럼 **기존 파일의 특정 부분을 찾아내어 수정하고, 오류 발생 시 대응**해야 하는 작업은 **Agent 모드**가 더 효율적이었을 수 있습니다. 반면, 아예 새로운 시스템을 처음부터 구축하거나 여러 파일 간의 의존성이 복잡했다면 **Plan 모드**가 더 안정적이었을 것입니다.
